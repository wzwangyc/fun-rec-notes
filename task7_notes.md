## 学习笔记：MTGR 混合范式建模

### 1. 纯生成式范式的代价
HSTU 和 GenRank 追求序列建模的纯粹性，但美团团队发现，完全去除**交叉特征（Cross Features）**会导致性能大幅下降。
* **原因：** 生成式序列受限于严格的因果依赖，无法在预测位置 $i$ 时“提前看到”候选物品的属性并进行统计交叉。
* **现状：** 即使增加模型规模（Scaling），也难以弥补由于信息缺失带来的性能缺口。

### 2. MTGR 的混合范式：生成架构 + 判别目标
MTGR 提出了一套“既要效率，又要精度”的方案：
* **核心思想：** 区分手段（Transformer 序列聚合）和目的（对候选进行判别式评分）。
* **数据组织：** 将同一用户的多个候选聚合，序列由 User + Seq + RealTime + Candidate Tokens 组成。
* **计算优势：** 保持了用户粒度的计算复用，m 个候选并行处理，复杂度由 $O(m \cdot n^2)$ 降至 $O((n+m)^2)$。

### 3. 架构创新：让异构特征和谐共处

#### 特征到 Token 的映射
MTGR 对 Token 进行了非对称组织：
* **历史 Token：** 仅编码历史，不依赖候选信息。
* **候选 Token：** 融合了物品固有特征与**交叉特征**。
  $$\text{CandidateToken}_i = \text{MLP}(\text{Concat}(\text{Emb}(\text{Item}_i), \text{Emb}(\text{Cross}_i)))$$

#### 分组归一化 (Group Layer Normalization)
由于不同类型的 Token（如用户属性 vs. 行为序列）处于不同的语义空间，分布差异巨大，传统的 LayerNorm 会导致语义干扰。
* **GLN 方案：** 将 Token 分为 User、Seq、RealTime、Candidate 四组，每组独立计算均值和方差。
* **价值：** 实现了分布对齐和语义独立，保证了混合建模时的训练稳定性。

### 4. 动态掩码 (Dynamic Masking)
为了在聚合样本中严格遵守因果性并防止信息泄露，MTGR 设计了三条动态掩码规则：

| 规则类型 | 作用对象 | 可见性逻辑 |
| :--- | :--- | :--- |
| **规则 1：全局可见** | User & Seq tokens | 静态长期历史，对所有后续 token 可见 |
| **规则 2：因果可见** | RealTime tokens | 严格按时间戳决定：只有发生时间早于目标的才可见 |
| **规则 3：相互屏蔽** | Candidate tokens | 候选之间不可见（对角线 Mask），确保独立评分 |

### 5. 总结：原理驱动的取舍
MTGR 的成功在于它认识到：
1. 用户粒度建模的本质是**计算复用**，而非必须采用自回归生成。
2. 好的归纳偏置（如交叉特征、GLN）比盲目追求模型规模更重要。
