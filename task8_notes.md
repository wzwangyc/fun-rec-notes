## 学习笔记：RankMixer硬件效率优化

### 1. 核心矛盾：为什么推荐模型在 GPU 上跑得慢？
传统推荐模型（DLRM）在 GPU 上的 MFU 通常只有 5%，远低于 LLM 的 45%。
* **访存瓶颈:** 操作以 Memory-bound 为主，访存量远大于计算量。
* **碎片化计算:** 计算图由多个小模块组成，Kernel Launch 开销巨大。
* **利用率低:** 无法充分利用 Tensor Core 的矩阵乘法加速能力。



### 2. RankMixer 的三大硬件感知设计

#### Token Mixing：更高效的信息融合
RankMixer 认为推荐系统需要的是信息混合而非基于相似度的动态加权。
* **复杂度飞跃:** 将 Self-Attention 的 $O(T^2D)$ 降低为 $O(TD^2)$。
* **计算逻辑:** 在特征维度而非 Token 维度进行混合。通过多层堆叠，同样能捕获高阶特征交叉。
* **硬件优势:** 将操作统一为大矩阵乘法，大幅提升计算密度。



#### Per-Token FFN：捕捉特征异质性
推荐系统的特征（如用户 ID、类目、统计点击率）语义空间完全不同。
* **参数隔离:** 每个 Token 拥有独立的 FFN 参数，避免不同分布的特征相互干扰。
* **学习效率:** 高信息量的 Token 可以分配更大的参数量，提升了参数利用的针对性。
* **FLOPs 守恒:** 虽然参数量增加了，但计算复杂度与共享 FFN 保持一致。

#### Sparse MoE：参数高效扩展
为了在不增加推理成本的前提下扩展参数量，RankMixer 引入了 MoE。
* **ReLU Routing:** 允许每个 Token 激活不同数量的 Expert，实现动态稀疏。
* **DTSI-MoE (Dense-Training / Sparse-Inference):** 训练时让更多 Expert 得到更新，推理时只激活少数，兼顾了充分训练与高效推理。



### 3. 硬件效率的终极飞跃
RankMixer 成功的关键在于将核心操作全部转化为 **Compute-bound** 的大矩阵乘法（GEMM）。
* **操作统一:** 约 85% 的计算时间都花在矩阵乘法上，能满负载运行 Tensor Core。
* **系统效益:** MFU 从 5% 提升到 45%，让推荐模型能直接利用大模型领域的成熟加速工具链。

### 4. 总结
RankMixer 证明了只要针对推荐场景进行针对性定制，统一的类 Transformer 架构同样能跑出极致的硬件效率。这为推荐系统后续的 Scaling 扫清了工程障碍。
